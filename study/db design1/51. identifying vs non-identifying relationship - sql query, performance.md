* single pk pros: simple is best
1. api, url design simplicity
- non-identifying: comments/500
- identifying: /boards/10/comments/3
  - 2 identifiers, comment depend on board
2. development convenience and preventing mistakes
- non-identifying: deleteComment(commentId)
- identifying: deleteComment(boardId, commentNo)
3. oop and orm 
- non-identifying: comment has an id field. repository.findById(500L)
- identifying: using identifier class is needed

conclusion: single pk makes simple in api, business logic, data access

* index usage
- non-identifying: pk index(comment_id) + fk index(board_id)
- identifying: composite pk(board_id, comment_no)
